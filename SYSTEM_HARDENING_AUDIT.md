# System Hardening Audit

Updated: 2026-03-29

This audit reflects the current Jarvis codebase after the stabilization pass in this branch. It is intentionally conservative: items are marked fixed only when code was changed in this pass, otherwise they are deferred.

| Subsystem | Issue | Severity | Impact | Proposed fix | Status |
|---|---|---:|---|---|---|
| Router | Typed router generated placeholder provider keys like `time-skill-provider`, which did not match the real registry and muddied staged diagnostics. | High | Action payloads and downstream debugging could point at non-existent providers. | Map router domains to canonical provider keys in `model-router.ts`. | Fixed |
| Router | Heuristic routing logic existed in both `router-fallback.ts` and `orchestrator-provider.ts`. | Medium | Signal tables could drift and route command/misson flows differently. | Remove duplicate table and route direct orchestrator calls through shared fallback logic. | Fixed |
| Router | Unused model prompt constant and low-confidence constant remained after the routing rewrite. | Low | Dead code increased confusion during audit. | Remove stale router constants. | Fixed |
| Provider results | Providers used inconsistent status wording and ad hoc failure shapes. | Critical | UI/store logic and operator debugging could not reliably tell dry-run blocks from capability blocks, transport failures, or unavailable bridges. | Add normalized `status`, `trace`, and richer `failure` metadata to provider/calendar results; move providers onto shared result helpers. | Fixed |
| Governance enforcement | `checkSkillGovernance()` and `enforce()` both wrote audit entries for the same action. | High | Duplicate audit records inflated logs and made governance history noisier and less trustworthy. | Keep auditing in `enforce()` only. | Fixed |
| Governance enforcement | Capability blocking did not consistently account for `write` requirements. | High | Some write actions looked promotable in governance/readiness despite missing write capability. | Centralize capability evaluation in governance decision logic, including `write_files`. | Fixed |
| Audit trail | All staged actions were force-labeled with `Blocked (dry run)` by `stageAction()`, including governance audit entries and non-dry-run staged flows. | High | System action log surfaced false reasons and obscured real blockers. | Stop auto-appending dry-run text; let callers provide truthful summaries. | Fixed |
| Audit trail | Audit entries did not carry their staged action id back into the returned audit object. | Medium | It was harder to correlate audit-log reasoning with staged runtime actions. | Attach `stagedActionId` in `recordAuditEntry()`. | Fixed |
| Readiness engine | Several actions were reported as promotable after flag changes even though the code path still only stages. | Critical | Activation docs overstated what could safely go live. | Add explicit `not_implemented` blocker and use it for staged-only concierge, calendar write, memory write, builder, and orchestrator staging paths. | Fixed |
| Readiness engine | Activation/readiness contradictions were not surfaced anywhere. | High | Operators could not easily spot when `activation-state.json` drifted away from current code/readiness truth. | Add `detectActivationStateMismatches()` and surface mismatch notes through runtime snapshot system lines/issues. | Fixed |
| Runtime file reads | SAFE_ROOT read/JSON failures silently fell back to defaults. | Medium | Broken runtime files looked like “empty data” instead of parse or read failures. | Add high-signal warnings for read and JSON parse failures. | Fixed |
| Calendar provider | `ComposedCalendarProvider` used read-adapter selection for writes, which could route write staging into read-only ICS adapters. | High | Calendar write staging could hit the wrong backend narrative and confuse operator expectations. | Route composed writes to the local staging provider and keep read adapter selection separate. | Fixed |
| Calendar provider | ICS adapter advertised read availability even when neither remote ICS nor local ICS existed. | High | System/readiness docs could falsely claim ICS local was live before a file existed. | Check for local ICS presence in `describe()` and report `UNAVAILABLE` when no ICS source exists. | Fixed |
| Calendar provider | Merged calendar results were not explicitly sorted after fallback merge. | Low | Event ordering could appear unstable across adapters. | Sort merged, de-duplicated events deterministically. | Fixed |
| Mail provider | Gmail fetch/send failure handling mixed bridge absence, provider failures, and transport failures under ad hoc codes/states. | High | Debugging retryability and root cause was slower and less reliable. | Normalize Gmail result statuses and classify transport vs provider failures consistently. | Fixed |
| Concierge provider | Draft-reply and booking flows surfaced as generic staged actions without machine-readable reason. | Medium | Operators could not quickly distinguish DRY_RUN staging from capability or implementation gaps. | Normalize staged concierge results with `status`, `trace`, and notes. | Fixed |
| Speech/media providers | Speech/media providers mixed unavailable/provider error semantics and did not return uniform traces. | Medium | Troubleshooting missing bridge vs invalid credential vs transient API failure required code inspection. | Move speech/media onto shared result helpers and consistent failure mapping. | Fixed |
| Memory selector | Documented module bias existed only in comments; score calculation treated no-domain-filter requests too evenly. | High | Memory ranking could underweight the most relevant domain for the requesting module. | Apply module bias directly in `scoreDomainMatch()`. | Fixed |
| Memory selector | Tag matching and tokenization were not normalized or de-duplicated. | Medium | Repeated tags/tokens and casing differences skewed scores and filters. | Normalize case, de-duplicate query tags/tokens, and stabilize sorting. | Fixed |
| Memory store | Query tag filtering was case-sensitive and merged persisted/receipt records without de-duplication. | Medium | Relevant records could be missed and duplicate ids could leak into results. | Make tags case-insensitive and de-duplicate merged records by id. | Fixed |
| Builder provider | Builder heuristics lived inside the provider, making them harder to test independently. | Medium | Builder remediation/decomposition logic was coupled to a heavier provider module than necessary. | Extract pure heuristics into `builder-heuristics.ts` and test them directly. | Fixed |
| Builder provider | Builder staged results did not distinguish designed staged-only behavior from dry-run staging. | Medium | Activation planning could not tell “not wired yet” from “wired but blocked by DRY_RUN.” | Normalize builder staged results and add staged-only readiness blockers. | Fixed |
| Runtime/System truth surfacing | Runtime snapshot lacked readiness summary and activation mismatch notes. | Medium | System tab/system ops surfaced partial truth only. | Include readiness summary and activation-state mismatch notes in runtime snapshot lines/issues. | Fixed |
| Provider registry | Registry remains eager and singleton-based. | Low | Harder to isolate providers in unit-style tests without additional indirection. | Leave current wiring in place; revisit only if provider count or startup cost grows materially. | Deferred |
| Calendar writes | No live Google/OpenClaw calendar write bridge exists in the current provider implementation. | Critical | Calendar writes cannot honestly be promoted yet, even with flags/credentials/governance satisfied. | Implement a real live write execution path, then add smoke tests on a credentialed machine. | Deferred |
| Memory writes | Structured memory writes still stage actions only; no SAFE_ROOT write bridge exists. | High | Wave-3 style “live memory write” activation would be misleading today. | Add a safe write bridge and persistence layer guarded by `DRY_RUN` and `CAPABILITIES.write`. | Deferred |
| Concierge live draft/booking | Draft generation and booking dispatch still stage only. | High | Activation docs cannot treat them as live-capable today. | Add live execution bridges and service integrations, then promote with targeted smoke tests. | Deferred |
| Builder live execution | Builder provider remains intentionally staged-only despite bridge primitives existing elsewhere. | High | Builder activation is not ready without a dedicated live provider path and operator guardrails. | Wire the live bridge into the provider and re-audit before any activation attempt. | Deferred |
| Credentialed/runtime verification | This pass did not execute real Gmail, GCal, Anthropic, ElevenLabs, or OpenClaw live calls. | Critical | First activation on a new machine can still fail on auth, scopes, or environment drift. | Run credentialed smoke tests on the target machine with network/execute/write enabled incrementally. | Deferred |
